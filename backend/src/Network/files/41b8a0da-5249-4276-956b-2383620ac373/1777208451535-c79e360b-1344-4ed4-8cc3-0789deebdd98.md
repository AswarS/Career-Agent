# KV Cache优化想法
## 基于token稀疏性的KV Cache压缩
### 目前的一些问题
- 1. 如何选取高hit的kv Cache
    - H2O：贪心策略
    - SnapKV：模型在decoding前就可以确定哪些token是high hit的，
- 2. 长输出时容易丢失历史信息（原本低hit的kv可能在某些地方很重要）
    - Quest:通过pagedAttention的min/max key与curr query找到upper bound，取对应top k个page作为cache
- 3. 部分task中prefill阶段的稀疏性可能并不明显（推理任务）
    - 不能在prefill阶段过度压缩
### 一些想法
- 1. attention score的top k采样
    - 以往类似H2O是对当前query选择attention score较高的进行evict，是基于“现在更关注该token”进行选择
    - 能否改成统计generate时到目前为止哪些kv多次得到较高的attention score，选择这些kv进行保留，是基于“generate时更关注该token进行选择”
    - 优点：可能能更好地筛选关键信息
    - 缺点：需要额外存储attention score，且需要额外计算hit rate，没有解决丢失信息的问题
- 2.  类似SCOPE，采用双阶段框架，分别对prefill的prompt和decoding进行压缩
    - 以推理任务为例，在decoding阶段中可能产生某些重要推论，也可能是一些无效的/错误的结果，或是对prompt的重复，这部分token也存在稀疏性
- 3. 信息丢失问题
    - 我认为丢失的信息应该主要是prompt里的信息，也就是prefill稀疏化引起的，decoding阶段的稀疏化可能不太会引起这个问题
    - 在稀疏化的基础上只针对prompt的kv cache做召回，召回的方法参考Quest，减少计算量和存储
### 思路总结
- 采用类似SnapKV的方法对prefill阶段进行压缩，得到prefill的cache池
- 在decoding阶段使用基于高注意力分数出现频次的top k采样方法进行evict，维护decoding的cache池
- 当某个query对cache池中的kv的注意力分数都不太高时，对完整prompt的kv做召回

## 基于重计算的优化方法
- 代表论文：REFORM，
    - 第一阶段：在不保留完整 KV Cache 的情况下，快速扫描长输入并提取关键信息用于后续检索
        - 循环分块处理：将超长输入分割成多个块（例如 32k tokens 一块）。迭代地处理这些块。在处理当前块时，基于前一个块的压缩KV Cache进行条件计算。
            - KV Cache 压缩：使用基于注意力的token驱逐策略（类似于 H2O 方法），只保留最重要的token，从而保持低内存占用。
            - 位置 ID 重分配：重新分配位置ID，使压缩后的 Cache 在逻辑上连续，允许模型处理超出预训练限制的长度。
        - 跨层上下文嵌入提取：REFORM 不直接使用压缩后的 Cache 进行生成，而是用它来构建轻量级的 Token 级嵌入。系统会收集来自特定层和特定注意力头的 QKV 状态，用于后续的相似度检索 。
        - 早退策略：
            - 研究发现，用于检索的高性能嵌入通常位于模型的中低层。
            - 因此，在提取完所需层的嵌入后，REFORM 会停止后续高层的计算。这显著减少了计算量和内存需求，因为不需要为高层保留KV Cache。
    - 第二阶段：构建检索嵌入
        - 基于 QKV 状态的余弦相似度：论文发现，直接使用注意力层输出的 QKV 状态进行余弦相似度搜索，其检索性能往往优于传统的注意力分数或隐藏状态。这种方法内存占用极小（仅需保存选定头的向量），但效果很好 。
        - 注意力头选择：并非所有注意力头都适合检索。作者通过合成任务（模式匹配和多跳问答）筛选出表现最好的头。
        - 最终的嵌入是将这些选定头的检索分数进行归一化和组合（在数学上等同于对各头独立计算相似度后取平均）。
    - 第三阶段：按需 Cache 重计算
        - 重要性识别：使用 Query（输入的最后一部分，即当前问题或指令）的嵌入与之前存储的上下文嵌入进行余弦相似度计算。通过最大池化（Max-pooling）和相邻 Token 平滑处理，识别出与 Query 最相关的上下文片段 。
        - Gather & Recompute：根据相似度分数，从原始输入中聚集出最相关的 Token（通常包括开头、结尾以及中间检索到的高分片段）。将这些选出的原始 Token 再次输入模型，进行完整的 KV Cache 计算 。
        - 生成：使用这个重新计算生成的、高质量且包含关键信息的 KV Cache 进行后续的文本生成 。
### 存在问题
- 对筛选出的token重新进行forwarding，成本可能较高
- 仅针对prefill阶段进行一次gather和recompute，没有利用decoding阶段的信息
### 一些想法
- 1. 筛选后的token可能并非都是关键信息，可能存在一些冗余，能否按重要性进行二次筛选，重要性低的复用之前的KV，重要性高的再重计算（个人认为这个方法不行）
- 2. 对decoding阶段进行扩展，将REFORM变成一种动态检索增强生成
    - 主要挑战：
        - 1. 推理延迟：如果频繁触发检索-加载-重计算这个过程可能会在推理过程中产生很高的推理延迟
        - 2. 计算成本：如果在decoding阶段频繁触发REFORM的逻辑，生成速度会变得极慢，因为每次更新上下文都需要重新forward一遍选中的片段。
    - 解决：
        - 在decoding 阶段设置触发器（例如每生成 512 个token，或者当模型困惑度升高时），利用当前生成的作为新的Query，对检索库进行更新
        - 为了不扩大 KV Cache，可以维护一个固定大小的“活跃 Cache 窗口”，当新检索到的片段需要进入Cache时，根据attention score，驱逐掉旧的KV Cache
        - 重计算token时冷冻住原先的KV Cache，只对evict进来的token进行重计算。
