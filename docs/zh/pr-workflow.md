# PR 工作流

## 目标

固定后续前端提交的 PR 流程，减少遗漏，让你作为中文审查者更容易判断是否可以合并。

## 默认流程

### 1. 本地完成改动

要求：

- 改动范围清晰
- 与当前 slice 对齐
- 不夹带无关改动

### 2. 本地验证

至少执行：

- `npm run build`

如有页面行为改动，还应补充必要的手动检查。

### 3. 提交并推送分支

要求：

- commit 信息简洁
- 分支名使用 `codex/` 前缀

### 4. 创建 PR

要求：

- 使用仓库内的中文 PR 模板
- 初始可以先用 draft，也可以直接 ready

### 5. 切换为待审查状态

如果 PR 一开始是 draft，完成自检后应切换为 `Ready for review`。

目的：

- 让 Copilot 和人工审查都在同一个正式状态下工作

### 6. 等待 Copilot 自动审查

通常需要 `5` 分钟以上。

要求：

- 不要在 PR 创建后立刻通过评论或命令主动召唤 Copilot
- 先等待自动审查进入队列并返回
- 不要在 Copilot review 返回前就直接请求你合并

推荐等待窗口：

- 先等待约 `5-10` 分钟
- 再检查一次 PR review 状态

如果超过这个窗口仍然没有返回评论，再进入补救动作：

- 确认 PR 已经是 `Ready for review`
- 确认 GitHub 不是单纯排队延迟
- 只有这时才允许主动请求 Copilot reviewer

### 7. 检查 Copilot 评论

处理原则：

- 真实 bug、状态同步、隔离、安全、可移植性问题：应优先修
- 纯风格建议：可以不采纳
- 不采纳的建议需要有明确理由

### 8. 修正后再次验证

至少再次执行：

- `npm run test`（从 Phase 5 开始）
- `npm run build`

### 9. 直接合并或告知结果

最终汇报时应说明：

- 本次改了什么
- Copilot 提了哪些有效问题
- 哪些问题已修
- 哪些建议未采纳以及原因
- 当前是否已经合并

当前默认执行方式：

- 每个 PR 只等待 1 轮 Copilot 审查
- 修正这一轮里的有效问题后，不再等待第二轮 Copilot
- 如无阻塞问题，Codex 可直接 `squash and merge`
- 合并后再向你同步结果和下一阶段判断

## 当前默认规范

- PR 用中文写，方便你审查
- Copilot review 返回 1 轮后即可收口
- 如果自动 review 在合理等待窗口后仍没触发，再由 Codex 主动通过 `gh` 请求 Copilot 审查
- 如无阻塞问题，修完有效评论后由 Codex 直接合并

## Codex 执行约束

为了避免流程在 git 中间节点异常中止，后续默认遵守：

- `git checkout -b`
- `git add`
- `git commit`
- `git push`

这些都视为中间步骤，不应作为单独停点。

默认只有两种情况可以暂停：

- 已经完成 `PR -> Copilot 1 轮审查 -> 修正 -> squash merge`
- 出现真实阻塞，例如权限、冲突、审查内容矛盾、或无法继续判断的技术问题

如果只是完成了 branch / stage / commit / push 其中某一步，Codex 应继续往下推进，而不是停住等待。

## 例外情况

以下情况可以直接说明并暂停：

- Copilot 没有返回评论
- 评论内容互相冲突
- 评论要求越过前端仓库边界去改后端或 agent 逻辑
- 评论含糊到无法判断是否该改
